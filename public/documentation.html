Overview

UGPlayground is an interactive playground where the user customizes a character, authors small artifacts (moves and shapes) inside dedicated labs, and invokes those artifacts back in the world by typing their names. The site is organized around one mental model: everything you author has a name; everything named can be invoked.

This page documents three slices of the system:

  • User — what you click and type, what the labs do, what each affordance produces.
  • System — how it's structured, why those choices were made, how a single keystroke threads through every layer.
  • Reference — a lookup-shaped catalog of types, hooks, components, and project terms.

1. User

1.1 Playground

Welcome to the playground! This is your world, this is your character. You get to teach your character how to do and make things with commands.

1.2 Customize your character

Press the Customize button in the bottom-right of the playground. Use the arrows and buttons to choose an animal, a color, and a biome. You can change this at any time in the customize menu – just press save.

1.3 Commands

The way you talk to your character is in the bottom Command Bar. Try typing wave and pressing the spacebar or enter.

Try another — type jump.

Both wave and jump are artifacts of the Dance Lab. They already learned these moves, and they can learn many more.

Chaining commands

You can chain commands together with a period. Try typing jump.wave. The character does each in sequence — jump first, then wave.

1.4 Labs

Last section we showed you how to type commands. Now we'll show you how to use these labs to write commands that give your character new skills. Every lab contains a library with a list of named artifacts you've written. Wave and jump are both Dance Lab artifacts — commands that make the character move.

Let's jump in and take a look at the Dance Lab.

Dance Lab

The Dance Lab is designed for writing dance moves. The lab gives you grabbable nodes on each limb; bend a limb by dragging the node at its middle. The green dot at the character's center is a force handle — drag it in a direction to apply force, sending the character moving that way.

Press Play to preview the move you've created. When you're satisfied and the name field has a value, press Save Move. Remember: every artifact needs a name so you can call it on the playground.

You can edit any saved move by clicking the pencil icon in the move library. You can play a saved move with the play button, or delete it with the X.

After saving a new move, return to the Playground and type your new command to see your character perform it.

Shape Lab

The Shape Lab lets you build shapes out of a small language of three letters:

  • T opens a new polygon.
  • S is a side.
  • B closes the polygon.

Let's do an example. This is how you write a triangle:

TSSSB

This is how you write a square:

TSSSB

Instead of writing S for a plain side, you can write another T…B block that makes that side itself a polygon. That's how you build compound shapes. Here's how you write a star — a polygon whose sides are themselves smaller polygons:

TTSSSBTSSSBTSSSBTSSSBB

This is what's called a type-2 grammar — a small fixed alphabet whose rules let you build endlessly nested compositions.

For now, the polygons you save are stickers your character can place on the playground when you type their name. Eventually they'll be interactive — you'll be able to build with them and create a world of your own.

2. System

The system layer is the architecture behind the user surface. How files are organized, how data flows from edit to render, how new modules plug in.

2.1 Commands

Picking up from User §1.3 — when you press space in the command bar, this is what happens under the hood:

  1. A keystroke lands in CommandInput — a controlled React component whose value lives in useCommand. Space calls submit().
  2. useCommand owns the input string, the green/red flash state, and the self-pruning history list. submit() trims the string and hands it to resolveCommandChain.
  3. resolveCommandChain is a pure function in data/command.ts. It splits on dots, looks each segment up in the move and shape libraries, and returns a CommandToken[] — a discriminated union of { type: "move", move } and { type: "shape", shape }. Pure because nothing in the resolver depends on time, React, or the DOM — that's why it sits in data/ while the surrounding hook sits in engine/.
  4. Validation is all-or-nothing. A failed segment returns null; the input flashes red and the entry logs as a strikethrough. A clean chain flashes green and the tokens go to useMoveSequencer.runChain.
  5. The sequencer dispatches per token type. A move fires its impulse through physics and cues articulation. A shape fires the sticker-spawn callback — because there is no associated animation, shape tokens are zero-cost to the metronome and take no time.

2.2 Lab modules

A lab is a workspace where the user authors an artifact — a named thing they save to a library and invoke from the playground. The Dance Lab and the Shape Lab are the two labs shipped today; they look different on the surface but they share the same skeleton. That shared skeleton is what makes new labs cheap to add.

Each lab provides four things:

  • Artifact type — a pure data shape describing what a saved thing looks like. A Move carries a pose, a force vector, and a name; a Shape carries a grammar string and a color seed.
  • Editor view — the lab's React component, where the user composes and names the artifact. The editor holds a draft of the artifact in component state; clicking Save commits the draft into the library.
  • Library — an array of saved artifacts persisted in local memory under a versioned key. The playground reads it on mount and the command resolver uses it to interpret typed input.
  • Command-resolver hook — the route back into the playground. Typing the artifact's name in the command bar invokes it (see §2.1 for the pipeline).

Why labs are extensible

Adding a new lab is about filling in the four slots. The engines (physics, articulation, sequencer) are shared across every lab; in many cases, the existing engines are equipped with whatever the new artifact invoke. There is no per-lab plumbing that has to be re-routed every time. The shipping cost of "another lab" stays roughly constant as the project grows.

The exception — and the seam for future growth — is when a lab introduces a new dynamic the existing engines don't cover. The current engines only know how to invoke transient events: a move fires once and ends; a shape spawns once and the sticker stops being engine-relevant. They have no concept of persistent world-loop interaction.

A planned future addition is a collision engine — giving labs access to what we'd call a simulation socket alongside the current invocation path. A lab using the simulation socket would let its artifacts persistently change the game loop of the world (collision, attraction, decay, anything that reads from and writes to the world every frame). The four-slot recipe stays the same; the difference is that the new socket changes how the engines react to the artifact's presence, not just to its invocation. That's the next layer of plumbing to add — and the only kind of plumbing the four-slot recipe doesn't already cover.

2.3 Engines

Picking up where §2.1 left off — once the sequencer has the resolved CommandToken[], three engines do the rest.

Move sequencer (useMoveSequencer)

The move sequencer closes the loop on the command pipeline. It reads a chain of resolved command tokens and queues physics and articulation per link.

  • For a move token, it fires the impulse through physics.applyImpulse and hands articulation a target pose, rotation, and duration.
  • For a shape token, it fires the spawn callback — a sticker drops at the character's current position. Shape tokens are zero-cost on the timing: they drain instantly and the next link fires.

Two clocks govern the link cadence. A fixed metronome — each move link occupies MOVE_TEMPO / speed seconds before the next link fires — keeps chains rhythmic. A stretched visual clock handles the launch case: when a move's impulse sends the character airborne, the sequencer predicts the airtime and hands articulation that longer window so a flip's rotation rides the arc. A live-altitude check completes the flip the moment the character actually lands.

Physics (useCharacterPhysics + stepPhysics)

Physics owns the character's position and velocity. It lives in two parts:

  • stepPhysics(state, intent, dt, world) is a pure function — no React, no side effects. Given the current physics state (position and velocity), the user's intent (move left, move right, jump), a time delta dt (seconds since the last tick), and the world parameters (gravity, friction, bounds, ground line), it returns the next state.
  • useCharacterPhysics is the React hook that wraps stepPhysics. It holds the live state, calls stepPhysics every animation frame (~60 times per second), and publishes the latest snapshot to the renderer.

The split exists so the heavy lifting (the physics math) can be tested without React — call stepPhysics with example inputs in a unit test and check the output. It also means new physics behaviors (collidable rectangles, friction zones, etc.) are added by changing the world data passed in, not by rewriting the function. The ground is one hardcoded plane at y = 0 today; the planned collision engine generalizes that into a loop over collidable rectangles, threaded through the same world parameter.

Articulation (useCharacterArticulation)

Articulation owns the character's pose and rotation. It tweens limb positions and whole-body rotation from their current values to the target the sequencer hands it, over a duration the sequencer also hands it. It knows nothing about gravity, velocity, or chain timing — its only contract is "tween these limbs and this rotation over this many seconds." Two invariants matter: tweens always start from the current value (so an interrupted move interpolates smoothly), and rotations snap forward (so an interrupted flip finishes its turn rather than unwinding back to start).

Camera (Playground.tsx + projection.ts)

Not an engine — a stateful transform that converts world coordinates to CSS pixels. Two pieces:

  • Playground.tsx computes cameraX (horizontal follow, clamped at world edges) and cameraY (vertical follow with a deadzone — ordinary jumps don't bob the camera, only real launches climb past the threshold).
  • projection.ts exports projectWorldToScreen(point, camera), used by both the character render and the StickerLayer.

2.4 End-to-end example: a typed command

Putting it together — one keystroke threads every layer above:

  1. A keystroke lands in CommandInput, a controlled input whose value lives in useCommand. Space calls submit(). (§2.1)
  2. submit() trims the input and hands the string to resolveCommandChain. The resolver splits on dots and looks each segment up in the move and shape libraries — the same libraries the labs wrote to (§2.2). A failed segment flashes red and logs a strikethrough; a clean chain flashes green and the resulting CommandToken[] goes to useMoveSequencer.runChain.
  3. The sequencer dispatches per token type. A move token fires its impulse through physics.applyImpulse and cues articulation with a target pose, rotation, and duration. A shape token fires the onSpawnSticker callback; the queue drains to the next link immediately. (§2.3)
  4. useGameLoop ticks at ~60Hz. Physics advances position and velocity via stepPhysics; articulation tweens pose and rotation toward their targets. Each engine publishes a fresh snapshot per frame.
  5. PlaygroundView sums the engine outputs each render. Playground.tsx projects the character's world coordinate to CSS pixels through projectWorldToScreen — the same helper the StickerLayer uses, so character and stickers stay locked to the same world frame. Character paints.
  6. When the sequencer's metronome slot elapses, the next link in the chain fires. If the previous move launched the character airborne, the live-altitude check completes the flip's rotation the instant the character actually lands — even if the predicted airtime was off.

One typed command, one path through every section of this page.

3. Reference

Lookup-shaped material — types, hooks, components, file locations, project terms. Use this when you know what you're looking for and need to know where it lives.

3.1 Types

Every artifact type lives in src/data/. Pure shapes; no React, no engine coupling.

Avatar
src/data/avatar.ts — the user's character: { animal, adjective, biome }. Resolved into a rig, palette, and biome scene at render time.
CharacterPose
src/data/characters/types.ts — the body's form at an instant: limb states and mouth state. The articulation engine tweens between poses.
Move
src/data/labs/dance-moves/dance.ts — the Dance Lab's artifact: { id, primary, secondary?, speed, targetPose?, forceVector?, rotationY?, rotationZ? }. Identified by command names; played by the sequencer.
Shape
src/data/labs/shapes/grammar.ts — the Shape Lab's artifact: { id, name, source, colorSeed }. The grammar source is the source of truth; geometry recomputes from source at render time.
Sticker
src/data/labs/shapes/stickers.ts — a placed Shape in the world: { id, shape: Shape, worldX, worldY }. The Sticker carries its own id (so the same Shape can be placed multiple times) and a frozen snapshot of the Shape it came from, so later edits to the underlying Shape don't mutate already-placed Stickers.
CommandToken
src/data/command.ts — the resolver's output: a discriminated union of { type: "move", move } and { type: "shape", shape }. Lives at the data root because it bridges both labs; the sequencer dispatches by type.

3.2 Engine hooks

Every stateful hook lives in src/engine/. They know about types and pure helpers, never about views.

useGameLoop
A thin requestAnimationFrame ticker. Calls onTick(dt) each frame; each engine subscribes independently.
useCharacterPhysics
The motion engine. Wraps the pure stepPhysics function in a frame loop and a snapshot. Exposes position, resting, setMovingLeft/Right, jump, applyImpulse, teleport.
useCharacterArticulation
The form engine. Owns pose and rotation tweens. Exposes currentPose, currentRotation, startMove, finishEarly, reset, isAnimating.
useMoveSequencer
The conductor. Accepts CommandToken[] and cues physics + articulation per link, dispatching by token type. Owns the two clocks (sequence metronome, visual duration).
useCommandRunner
The typed-command input stream. Owns the input string, submit logic, flash state, and the self-pruning history list. Resolves chains via resolveCommandChain and hands tokens to the sequencer.
useKeyboardInput
Maps keydown/keyup events to direct character intent callbacks (left, right, jump). A disabled flag silences it when the command input is focused or customize mode is open.
useLocalStorage
Typed persistence. Same signature as useState — [value, setValue]. Reads once on mount; writes through on update. The editor/playground bus.

3.3 Component hierarchy

The top-level App routes between three views by a single currentView state. Only one view mounts at a time; the others are torn down completely. HamburgerMenu is scoped to the playground branch — it mounts alongside the playground and unmounts when a lab is opened, so the slide-in nav is only present when there is a playground behind it to slide over. 

App
├── Playground branch              (currentView === "playground")
│   ├── HamburgerMenu              (fixed top-right toggle + slide-in nav)
│   ├── PlaygroundView
│   │   └── Playground             (camera + projection)
│   │       ├── Biome              (sky + parallax layers)
│   │       ├── StickerLayer       (placed shapes, projected)
│   │       ├── Character          (animal SVG + pose paths)
│   │       ├── WelcomeBanner      (first-load greeting)
│   │       ├── CustomizeAvatar    (slide-over panel)
│   │       ├── CommandInput       (fixed bottom-center)
│   │       └── CommandHistory     (fixed bottom-left)
│   └── LabSelection               (lab cards, bottom strip)
├── DanceLab                       (currentView === "danceLab")
│   ├── MoveLibrary                (saved moves list)
│   ├── Character (preview)        (live ragdoll)
│   ├── PoseHandles                (limb drag handles)
│   ├── ForceHandle                (force vector handle)
│   └── MoveForm                   (name/speed/rotation fields)
└── ShapeLab                       (currentView === "shapeLab")
    ├── ShapeLibrary               (saved shapes list)
    ├── ShapeRenderer (preview)    (live polygon preview)
    └── ShapeForm                  (grammar/name fields)

The full-unmount-on-view-switch is the architectural guarantee that backs the library-freshness invariant in §2.5.

3.4 File map

Top-level src/ structure, with one line per folder.

src/data/
Pure types and helpers. characters/ is reserved for character aesthetics (pose types, palette, animal rigs). labs/ groups per-lab artifact data — labs/dance-moves/dance.ts for moves, labs/shapes/grammar.ts for shape grammar, labs/shapes/stickers.ts for placed-shape artifacts. command.ts at the data root holds the cross-lab command surface (CommandToken, the resolver, the cross-library collision check). Plus biomes/ (scenes) and standalone files (avatar.ts, ids.ts).
src/engine/
Stateful hooks. The game loop, the two engines, the conductor, the input stream, persistence, keyboard mapping. Plus projection.ts — the shared world-to-screen camera helper.
src/components/
Views. playground/ holds the playground view and its overlays (Playground, PlaygroundView, Biome, CommandInput, CommandHistory, StickerLayer, WelcomeBanner). Per-lab editors live in labs/dance-lab/ and labs/shape-lab/ under a shared labs/lab.css for the common header/library/main chrome. Avatar customization lives in customize-avatar/. Top-level chrome shared across the playground branch sits directly under components/: HamburgerMenu (slide-in nav), LabSelection (the strip of lab cards beneath the playground), and Character (the animal SVG renderer used by both the playground and the dance lab).
public/
Static assets and standalone HTML pages (documentation.html, about.html). Served as-is by CRA.

3.5 Glossary

Artifact
A user-authored named thing — a Move, a Shape — that the playground knows how to invoke.
Socket
One of two paths an artifact can take to affect the playground: invocation (event-driven) or simulation (frame-driven).
Invocation
Engine touches the artifact only during dispatched events. Moves and shape-spawns are invocation-shaped today.
Simulation
Engine touches the artifact every frame, reconciling the character against it. Reserved for future collidable artifacts.
Draft / Commit / Enforce
The three-step contract between an editor and the playground: edit a working copy, write to storage, read from storage at the consumer.
The metronome
The sequencer's fixed clock — each chain link occupies MOVE_TEMPO / speed seconds before the next fires.
The visual clock
The sequencer's stretched clock for the articulator. Lets a launch's flip spread across the predicted airtime rather than the metronome beat.
The four-slot recipe
A new module needs an artifact type, an editor, a storage key, and a socket choice. See §2.7.
The dependency triangle
data/ → engine/ → components/, imports only down. See §2.2.
